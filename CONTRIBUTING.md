# Contributing

Contributions to the Engineering ADR Playbook are welcome.

## What kinds of contributions are useful

**Template improvements:** If you find the templates missing fields that are important for decision documentation, or if the guidance in a template is unclear, a pull request with a suggested change and an explanation of why it would help is welcome.

**New example ADRs:** Examples that demonstrate the ADR format for decision types not currently covered are useful. New examples should represent real decisions that teams face, not hypothetical scenarios constructed to demonstrate the format. Preference for decisions that involve meaningful trade-offs that the example documents clearly.

**Documentation improvements:** The docs in this repo explain when and how to use ADRs. If the guidance is incomplete or does not match how teams work in practice, specific suggestions for improvement are helpful.

## What this repo is not looking for

- ADRs for specific company or product decisions. The examples in this repo are generic enough to apply across different contexts.
- Process documentation that prescribes a specific toolchain or workflow for managing ADRs. The approach here is tool-agnostic by design.
- Content that promotes specific vendor products or services.

## How to contribute

1. Open an issue describing the proposed change before writing a pull request for significant changes. For small corrections (typos, broken links), a pull request directly is fine.
2. Pull requests should include a clear description of what changed and why.
3. For template changes, include an example of how the change affects an existing or new example ADR.
